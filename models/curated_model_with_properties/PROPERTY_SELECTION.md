# MPTA-Repair 性质选择说明

本文档说明 **MPTA-Repair**（Multi-Property Timed Automata Repair）精选数据集中每个时间自动机保留的性质含义，以及选择该性质的原因。筛选原则是：每个模型最多保留 3 条性质；优先选择能表达协议/系统关键需求的性质；尽量排除仅仅复述某个位置不变式或单条 guard 的浅层性质；所有性质均已通过 `verifyta.exe` 验证满足。

## BangOlufsen / 5bando

### P1

```text
A[] Sender_A.stop or Sender_A.end_jam or (A_c <= 28116)
```

含义：发送方 A 在协议执行过程中，若尚未到达 `stop` 或 `end_jam` 这类完成/退出阶段，则其局部时钟 `A_c` 必须不超过 `28116`。这表达了音视频总线协议中一次发送或碰撞处理过程不能无限期占用总线。

选择理由：该性质刻画了协议层面的全局有界响应，而不是单个位置的不变式。它反映工业协议中“发送过程必须在有限时间内完成或进入碰撞结束处理”的核心实时性要求。

### P2

```text
A[] not Sender_A.jam or (A_c <= 25000)
```

含义：当发送方 A 处于 `jam` 状态时，碰撞后的阻塞信号持续时间不能超过 `25000`。

选择理由：碰撞处理是该类总线协议的关键安全机制。选择该性质是因为它约束了碰撞后的总线占用时间，避免错误模型中出现 jam 信号过长导致总线长时间不可用。

### P3

```text
A[] not Sender_B.newPn or (B_c <= 40)
```

含义：发送方 B 在 `newPn` 状态中更新总线输出时，必须满足 `40` 时间单位内完成输出延迟。

选择理由：该性质对应物理总线输出延迟，体现了协议对硬件传播和采样时序的要求。它比普通可达性或死锁性质更能体现实时通信协议的物理约束。

## csma_cd / 2csma

### P1

```text
A[] not P1.sender_transm or not P2.sender_transm or (P1.x < 52)
```

含义：发送方 P1 和 P2 不能在超过碰撞检测窗口的时间内同时处于传输状态；如果二者同时传输，P1 的局部时钟必须小于 `52`。

选择理由：这是 CSMA/CD 协议的核心安全性质，表达“并发发送导致的冲突必须在有界时间内被发现”。相比单纯检查某个状态可达，它直接对应碰撞检测机制。

### P2

```text
A[] not P0.bus_collision1 or (P0.x < 26)
```

含义：总线自动机 P0 处于 `bus_collision1` 碰撞传播阶段时，碰撞信号必须在 `26` 时间单位内继续传播或被处理。

选择理由：该性质刻画介质传播延迟上界，是 CSMA/CD 正确性的基础。它连接了总线模型和发送方模型，比只检查发送方状态更能体现系统级时间约束。

### P3

```text
A[] not P1.sender_transm or (p2t == 0) or (P1.x < 52)
```

含义：当 P2 正在传输时，P1 若也处于传输状态，则 P1 的传输重叠时间必须小于 `52`。

选择理由：该性质通过共享变量 `p2t` 表达跨进程的互斥/冲突窗口，比简单位置约束更有实际意义。它能在后续错误注入中检测跨进程时序交互是否被破坏。

## Elevator / 3elevator

### P1

```text
A[] not Door.idle or (x>=2 and x<=5)
```

含义：门处于 `idle` 状态时，全局时钟 `x` 必须位于 `[2, 5]`，即门的稳定状态只能在规定时间窗口内出现。

选择理由：该性质表达电梯门动作的时间窗口约束，对应用户安全和机械动作完成时间。它比单纯死锁检查更贴近电梯系统的实际需求。

### P2

```text
A[] not Elevator.First_Floor or (x>=2)
```

含义：电梯到达或处于 `First_Floor` 状态时，必须已经经过至少 `2` 个时间单位。

选择理由：该性质刻画楼层移动的最小时间约束，防止模型出现“瞬间到达楼层”的不真实行为。它适合作为后续变异修复中的时间下界性质。

## engine / engine

### P1

```text
A[] not GearControl.GearChanged or ErrStat != 0 or UseCase != 0 or (SysTimer <= 1000)
```

含义：在无错误且正常工况 `UseCase == 0` 下，换挡完成状态 `GearChanged` 必须在 `1000 ms` 内达到。

选择理由：该性质直接对应工业需求“正常换挡应在 1 秒内完成”，是齿轮控制系统最重要的性能指标之一。它表达了真实需求，而非模型内部结构细节。

### P2

```text
A[] not GearControl.GearChanged or ErrStat != 0 or (SysTimer <= 1500)
```

含义：只要没有不可恢复错误，任意可恢复场景下的换挡完成时间都不能超过 `1500 ms`。

选择理由：该性质覆盖正常和异常但可恢复工况，约束范围比 P1 更广。它能检测错误注入后系统是否仍保持总体响应时间上界。

### P3

```text
A[] not GearControl.GearChanged or ErrStat != 0 or UseCase != 1 or (SysTimer <= 1055)
```

含义：当发动机未能及时提供零扭矩、但系统仍可恢复时，换挡完成时间必须不超过 `1055 ms`。

选择理由：该性质体现容错场景下的精细时间需求，能区分正常场景和特定故障场景。选择它可以让数据集包含更丰富的多性质组合，而不是只保留普通性能上界。

## FDDI / 8fddi

### P1

```text
A[] (ST1.x <= 140)
```

含义：站点 ST1 的令牌访问等待时间必须不超过 `140`。

选择理由：FDDI 的核心目标是保证令牌环访问时间有界。该性质代表站点响应时间上界，是令牌环协议中最基本的实时性要求。

### P2

```text
A[] (ST1.station_z_idle or ST1.station_y_idle) or (ST2.station_z_idle or ST2.station_y_idle)
```

含义：ST1 和 ST2 不能同时处于非 idle 的传输状态；换言之，至少有一个站点必须处于 idle。

选择理由：该性质表达令牌互斥访问，避免两个站点同时占用环路。它是 FDDI 协议安全性的核心，而非单个计时器约束。

### P3

```text
A[] not (ST1.station_z_idle or ST1.station_y_idle) imply (ST1.x <= 120)
```

含义：当 ST1 正在持有令牌并进行传输时，其连续占用令牌的时间必须不超过 `120`。

选择理由：该性质对应目标令牌旋转时间或令牌持有时间限制。它能检测错误模型是否破坏了公平访问和带宽分配。

## fischer / fischer_2_32_64

### P1

```text
A[] P(1).cs imply P(1).x >= 64
```

含义：进程 P(1) 进入临界区 `cs` 时，其局部时钟 `x` 必须至少达到 `64`。

选择理由：Fischer 协议依赖最小等待时间保证互斥。该性质表达进入临界区之前必须满足的最小延迟，是协议正确性的关键。

### P2

```text
A[] P(1).req imply P(1).x <= 32
```

含义：进程 P(1) 处于请求状态 `req` 时，其局部时钟不能超过 `32`。

选择理由：该性质表达请求阶段最大延迟，防止进程在请求阶段无限停留。它与 P1 共同形成 Fischer 协议中最小等待和最大请求延迟的时间约束组合。

## lynch / lynch_2_16

### P1

```text
A[] not P(1).L2 or (P(1).c <= T)
```

含义：进程 P(1) 在 L2 阶段的停留时间必须不超过协议参数 `T`。

选择理由：Lynch 互斥算法依赖阶段化等待保证进程间时序关系。该性质约束尝试区早期阶段，是互斥正确性的基础。

### P2

```text
A[] not P(1).L8 or (P(1).c <= T)
```

含义：进程 P(1) 在退出区 L8 阶段也必须满足时间上界 `T`。

选择理由：退出区有界完成关系到弱无死锁和资源释放。选择该性质可以覆盖临界区之后的行为，而不只关注进入临界区之前的阶段。

### P3

```text
A[] not P(1).L5 or (P(1).c <= T)
```

含义：进程 P(1) 在 L5 阶段的等待时间必须不超过 `T`。

选择理由：L5 表示尝试区后期判断与返回逻辑，该阶段的时间约束影响进程是否能正确竞争临界区。它补充了 P1/P2，使性质覆盖尝试区和退出区。

## mutex / mutex

### P1

```text
A[] not Ctrl.wait_for_s2 or (Ctrl_z <= 1000)
```

含义：控制器处于等待 S2 安全状态的阶段时，其轮询周期时钟 `Ctrl_z` 不能超过 `1000`。

选择理由：该性质表达 PLC 轮询控制器必须在周期上界内观察环境。它对应改进互斥协议中的等待延迟机制。

### P2

```text
A[] not Ctrl.g1 or (Ctrl_z <= 1000)
```

含义：控制器在 `g1` 授权阶段中，也必须保持 `1000` 以内的轮询周期。

选择理由：授权阶段如果观察过慢，会导致控制器误判被授权站点的安全状态。该性质反映互斥协议中授权监测的实时性。

### P3

```text
A[] not S1.I_am_safe or (S1_z <= 1000)
```

含义：站点 S1 处于安全状态时，其本地轮询周期时钟 `S1_z` 必须不超过 `1000`。

选择理由：该性质覆盖被控制对象自身的周期行为，与控制器性质形成互补。它有助于检测错误注入后局部站点周期是否被破坏。

## Peacemaker / 6pacemaker

### P1

```text
A[] not PURI_test.interval or (PURI_test.t >= TURI)
```

含义：在上限心率检测区间中，距离上一次心室事件必须至少达到 `TURI`，才能允许新的心室起搏。

选择理由：该性质表达起搏器最重要的临床安全约束之一：不能以过高频率刺激心室。它直接关系到病人生理安全。

### P2

```text
A[] not PAVI.AVI or (PAVI.t <= TAVI)
```

含义：房室间隔 AVI 状态下，必须在 `TAVI` 时间内发生心室感知或起搏。

选择理由：AVI 约束保证心房事件之后心室响应及时发生，是双腔起搏器保持 A-V 同步的关键。

### P3

```text
A[] not PVRP.VRP or (PVRP.t <= TVRP)
```

含义：心室不应期 VRP 的持续时间必须不超过 `TVRP`。

选择理由：VRP 用于过滤噪声和过早心室事件。选择该性质可以覆盖起搏器的噪声过滤机制，而不仅是起搏触发机制。

## Philips_Audio_Protocol / bocdp

### P1

```text
A[] not Sender_A.transmit or not Sender_B.transmit or (A_c <= 781)
```

含义：两个发送方不能在超过半 bit-slot 的时间内同时处于传输状态。

选择理由：该性质表达总线碰撞窗口，是音频协议中避免长时间冲突的关键安全需求。

### P2

```text
A[] not Sender_A.hold or (A_c <= 28116)
```

含义：发送方 A 在 `hold` 静默等待阶段中，必须在 `28116` 时间单位内结束等待。

选择理由：该性质对应 radio silence 要求，保证碰撞或消息结束后总线有足够但有界的静默间隔。

### P3

```text
A[] not Sender_A.jam or not Sender_B.jam or (A_c <= 25000)
```

含义：当两个发送方都处于碰撞后的 jam 相关状态时，jam 持续时间不能超过 `25000`。

选择理由：该性质约束碰撞处理阶段，防止错误模型造成总线持续阻塞。它与 P1/P2 共同覆盖传输、静默和碰撞恢复三个关键阶段。

## simop / simop

### P1

```text
E<> COM.COM4
```

含义：系统存在一条执行路径，使通信模块 COM 能到达 `COM4` 状态，即能够完成命令 0 请求之后的网络回复处理路径。

选择理由：原始 simop 模型没有给出性质。该性质不是简单位置不变式，而是验证 PLC、COM、NET、RIO 之间存在有效通信链路，能体现系统功能可达性。

### P2

```text
E<> RIO.RIO5
```

含义：系统存在一条执行路径，使远程 I/O 模块 RIO 到达 `RIO5`，表示它能够产生信号输出并返回对应回复。

选择理由：该性质补充 P1，覆盖远程 I/O 侧的响应行为。选择它是为了让 simop 不仅验证通信模块自身可达，也验证现场 I/O 设备能够参与完成一次有效交互。

## TarTarRepairedDB / 1repaireddb2

### P1

```text
A[] not client.serReceiving or (x <= 4)
```

含义：修复后的数据库模型中，客户端处于接收服务响应 `serReceiving` 状态时，响应时间必须不超过 `4`。

选择理由：该性质保留修复版本的核心有界响应需求，直接体现 repairedDB 的修复目标。由于该模型本身只有一条可满足核心性质，因此只保留这一条。

## train / trainahv93_2

### P1

```text
A[] not (controller.controller3 && cnt>0)
```

含义：控制器处于准备升起道闸的 `controller3` 状态时，计数器 `cnt` 必须为 0，即不能仍有列车在等待或通过。

选择理由：这是列车-道闸系统的核心安全性质，防止道闸在仍有列车时升起。

### P2

```text
A[] not (gate.gate1 && (train(1).train2 or train(1).train3))
```

含义：当道闸仍处于关闭动作 `gate1` 时，列车不能已经进入道口相关状态 `train2/train3`。

选择理由：该性质直接表达“列车到达道口前道闸必须完成关闭”的安全要求，比单独检查道闸关闭时间更具有系统意义。

### P3

```text
A[] not (train(1).train1 && gate.gate0) or (train(1).x <= 2)
```

含义：如果列车仍处于接近阶段且道闸是打开状态，那么列车时钟必须不超过 `2`，即列车仍处于安全接近窗口内。

选择理由：该性质将列车位置、道闸状态和时间窗口结合起来，能检测道闸响应是否足够早，适合作为后续错误注入后的修复目标。
